# Proyecto: Extracción Comparativa de Hemogramas

## Objetivo

Extraer datos de hemogramas desde dos fuentes distintas usando la estrategia más adecuada para cada una, comparar los resultados y generar un informe de métricas.

- **PDFs digitales de IDEXX** → `pdfplumber` (extracción directa de texto)
- **Fotos/escaneos de vouchers de Vargas Vet** → `EasyOCR` (reconocimiento óptico con post-corrección)

## `config.py` — Parámetros centralizados

Unifica todas las constantes del proyecto para evitar valores mágicos dispersos.

| Parámetro | Valor | Propósito |
|---|---|---|
| `COLUMNAS` | `["item","resultado","referencia","flag"]` | Columnas del hemograma |
| `OCR_CONFIANZA_MINIMA` | 0.2 | Umbral de confianza para EasyOCR (ignorar detecciones ruidosas) |
| `OCR_TOLERANCIA_Y_FACTOR` | 0.6 | Factor para agrupar palabras por fila (Y-clustering) |
| `OCR_GPU` | False | Usar CPU para EasyOCR |
| `PDF_TEXTO_MINIMO` | 80 | Si un PDF extrae menos caracteres, se considera escaneado |
| `EXTENSIONES_IMAGEN` | jpg, jpeg, png, tiff, bmp, webp | Formatos de imagen soportados |
| `MAPA_COLUMNAS` | — | Sinónimos de nombres de columna |

## `main.py` — Punto de entrada

### Flujo

1. **Detectar si `--graficos` está presente** en los argumentos. Si está, extrae los gráficos del PDF además del texto.
2. **Selector de archivos**: Si no se pasan argumentos, abre un diálogo `tkinter` para elegir archivos.
   - Si se pasa un archivo, procesa ese.
   - Si se pasa un directorio, procesa todos los archivos dentro.
3. **`detectar_tipo(archivo)`**: Clasifica automáticamente la estrategia:
   - `.pdf` → abre con `pdfplumber`, extrae texto de página 1.
     - Si el texto es ≥ 80 caracteres → **pdfplumber** (PDF digital IDEXX).
     - Si el texto es < 80 caracteres → **EasyOCR** (PDF escaneado).
   - `.jpg/.png/etc` → directamente **EasyOCR**.
4. **Procesar**: Llama a `extraer_pdf()` o `extraer_ocr()` según el tipo detectado.
5. **`--graficos`**: Si es pdfplumber y la flag está activa, llama a `extraer_graficos()` para detectar y recortar gráficos desde las páginas renderizadas.

### Uso

```bash
python main.py                               # Selector gráfico (tkinter)
python main.py data/pdfs/mi_archivo.pdf      # Archivo específico
python main.py data/                          # Directorio completo
python main.py --graficos data/pdfs/mi_archivo.pdf  # + extraer gráficos
```

## `extractor_pdf.py` — Estrategia pdfplumber

Extrae datos de hemogramas de PDFs digitales de IDEXX ProCyte Dx.

### Pipeline

1. **Abrir PDF** con `pdfplumber.open()`.
2. **Recorrer páginas**, extraer texto con `pagina.extract_text()`.
3. **`_obtener_lineas_limpias(texto)`**: Filtra líneas irrelevantes:
   - Encabezados del laboratorio ("1 888", "IDEXX", "pet owner:", etc.)
   - Fechas, pies de página, "Account #", "Generated by", etc.
   - Nombres de mascota que interfieren.
4. **`_parsear_lineas(lineas)`**: Procesa cada línea:
   - Separa tokens por espacios.
   - Detecta el primer token numérico → ese es el `resultado`.
   - Todo antes del número → `item`.
   - Lo que sigue → `referencia` y opcionalmente `flag` (H/L).
5. **Continuaciones**: Si una línea tiene una sola palabra no numérica, se fusiona con el `item` de la línea anterior (ej: "Reticulocyte" + "Hemoglobin" → "Reticulocyte Hemoglobin").
6. **`_separar_referencia_flag`**: Detecta si el último token es H/L y lo separa de la referencia (con protección para no confundir unidades como "fL", "g/dL" con flags).

### Por qué fallan otros enfoques

- **Tablas con `pdfplumber` (`page.find_table()`)**: Los PDFs IDEXX no tienen bordes de tabla dibujados como líneas; los datos están espaciados con espacios simples, no dobles. Las tablas con `vertical_strategy: "lines"` no detectan nada.
- **Regex puro**: Los nombres de los items (ej: "Reticulocyte Hemoglobin") contienen espacios, y los valores pueden tener rangos como "5.65 - 8.87 M/µL". Un regex para 4 columnas falla con items multi-palabra.
- **Enfoque token-based**: Detecta dónde empiezan los números y asigna columnas desde ahí. Es robusto porque en los PDFs IDEXX los items nunca contienen números.

## `extractor_ocr.py` — Estrategia EasyOCR

Extrae datos de imágenes de vouchers de Vargas Vet usando OCR.

### Pipeline

1. **Cargar imagen** con OpenCV.
2. **EasyOCR**: `reader.readtext(imagen)` devuelve lista de (bbox, texto, confianza).
3. **Filtrar por confianza** ≥ `OCR_CONFIANZA_MINIMA` (0.2).
4. **`_estructurar_tabla(resultados)`**:
   a. **`_agrupar_por_y(items)`**: Ordena palabras por coordenada Y, agrupa en filas cuando la diferencia de Y es ≤ 0.5× la altura promedio de las palabras. Esto maneja vouchers escaneados donde las filas no están perfectamente alineadas.
   b. **`_separar_filas_por_altura(filas)`**: Detecta palabras anómalamente altas (como la "H" de flag con 74px) y las separa en su propia fila si superan 1.8× la mediana de altura.
   c. **`_detectar_zonas(filas)`**: Encuentra los límites entre columnas de dos formas:
      - **Header detection**: Si una fila contiene palabras clave como "item", "resultado", "referencia" → usa los puntos medios entre las palabras del encabezado como cortes de columna.
      - **Gap analysis**: Si no hay encabezado, encuentra los 3 gaps más grandes entre coordenadas X de todas las palabras para definir las 4 columnas.
      - **Fallback**: Divide el ancho de la imagen en 4 partes iguales.
   d. **`_mapear_filas(filas, zonas)`**: Asigna cada palabra a la columna correspondiente según su X. Las filas después del header se convierten en filas de datos. Pasa por el corrector (`corregir_item()`, `corregir_resultado()`, `corregir_referencia()`).

### Dificultades con vouchers escaneados

- **Calidad variable**: Las imágenes tienen diferentes resoluciones (1086×1448 vs 1200×1600 px), fondos texturados, manchas, pliegues.
- **Alineación**: Los vouchers no están perfectamente cuadrados; pueden tener rotaciones leves.
- **Columnas sin bordes**: Las columnas están definidas solo por espaciado, y los espacios varían entre vouchers.
- **Caracteres confundidos**: EasyOCR confunde sistemáticamente:
  - "W" → "U" o "V" (UBC → WBC)
  - "M" → "H" (MID → HID)
  - Caracteres especiales como "µ", "^" se pierden.
  - Números: "1" → "I", "0" → "O", etc.

## `preprocesador.py` — Preprocesamiento de imágenes

### Pipeline

1. **Cargar en escala de grises**.
2. **`_detectar_angulo(gris)`**: Detecta rotación de la imagen usando la transformada de Hough sobre bordes Canny. Si el ángulo mediano de las líneas horizontales es > 1.5°, corrige.
3. **`_rotar(imagen, angulo)`**: Rota la imagen con `cv2.warpAffine` usando interpolación cúbica y `BORDER_REPLICATE` para no dejar bordes negros.
4. **CLAHE** (`cv2.createCLAHE(clipLimit=2.0, tileGridSize=(8,8))`): Mejora el contraste local. Reemplazó a un threshold adaptativo agresivo que destruía texto en fondos texturados.

### Por qué CLAHE y no threshold adaptativo

El threshold adaptativo (`cv2.adaptiveThreshold`) funcionaba bien en fondos uniformes pero destruía caracteres cuando:
- El voucher tenía una foto de fondo o trama de seguridad.
- La iluminación era desigual (sombra de un pliegue).
- El texto era muy fino (ej: "10^9/L" se partía en pedazos).

CLAHE preserva la legibilidad al mejorar el contraste local sin binarizar.

## `corrector.py` — Corrección post-OCR

### Estrategia

Mapa de correcciones hardcodeadas para errores sistemáticos de EasyOCR en este dominio específico (~50 reglas).

### Diccionarios

| Diccionario | Propósito | Ejemplos |
|---|---|---|
| `CORRECCIONES_ITEM` | Nombres de analitos mal OCRizados | UBC→WBC, HID#→MID#, PLI→PLT, RDV-V→RDW-CV, HCh→MCH, etc. |
| `CORRECCIONES_ITEM_PARCIAL` | Protege items que coinciden parcialmente con una clave | MCH, MCV, MPV, RBC — no deben corregirse |
| `CORRECCIONES_UNIDAD` | Unidades mal leídas | AL→fL, s/dL→g/dL, 10*9/1→10^9/L, coma→punto decimal |
| `PATRONES_ITEM` | Correcciones por regex | LYN→LYM%, MID%→MID%, GRA[vky]?→GRA% |

### Funciones

- `corregir_item(texto)`: Aplica `CORRECCIONES_ITEM` exactas, luego `PATRONES_ITEM` por regex.
- `corregir_resultado(texto)`: Aplica `CORRECCIONES_UNIDAD` sobre el resultado.
- `corregir_referencia(texto)`: Ídem sobre la referencia.

### Limitaciones

- No maneja **errores numéricos** (ej: "9.42" leído como "9.4?").
- Depende de que EasyOCR detecte el texto **con las letras correctas pero confundidas**; si el texto se omite completamente, no hay nada que corregir.
- No hay modelo ML — es un enfoque determinístico que cubre los casos más frecuentes observados en las 5 imágenes de prueba.

## `extractor_graficos.py` — Extracción de gráficos desde PDF

### Propósito

Los PDFs IDEXX contienen gráficos vectoriales (scatter plots de WBC, histogramas RBC/PLT) en página 2. No están incrustados como imágenes, por lo que hay que renderizar la página y detectar las regiones de gráficos.

### Pipeline

1. **Renderizar página** con PyMuPDF a 2× resolución.
2. **Binarizar**: Threshold a 230 (invertido).
3. **Morfología**: Cerrar + dilatar con kernel 15×15 para conectar las líneas finas de los gráficos en regiones sólidas.
4. **Contornos**: `cv2.findContours()` + `cv2.boundingRect()`.
5. **Filtrar regiones**:
   - Relación de aspecto < 3:1 (excluye bloques de texto anchos como encabezados)
   - Ancho ≥ 150, alto ≥ 120
   - Área entre 5000 y 200000 píxeles
6. **Fusionar rectángulos cercanos** en la misma fila horizontal.
7. **Recortar y guardar** cada región como PNG.

### Resultado esperado por PDF

- Página 1: ~3 gráficos (scatter plots del diferencial WBC)
- Página 2: ~2 gráficos (histogramas RBC y PLT)

## `normalizar.py` — Normalización para comparación

Usada por `metricas.py` para comparar el valor extraído contra el ground truth.

1. Limpia espacios, convierte a mayúsculas.
2. Colapsa espacios múltiples.
3. Intenta convertir a `float` para normalizar formatos numéricos (ej: "9,42" → 9.42 → "9.42").

Esto permite que "9.42" y "9,42" se consideren iguales, sin depender del formato exacto.

## `metricas.py` — Evaluación y generación de informe

### Pipeline

1. **Cargar ground truth** desde `ground_truth.json`.
2. **Para cada archivo**: ejecuta `procesar(archivo)` → obtiene el DataFrame extraído.
3. **Comparar fila por fila** contra ground truth usando `normalizar()` en cada columna.
4. **Acumular métricas**:
   - `col_item`, `col_resultado`, `col_referencia`, `col_flag`: cuántas filas acertaron esa columna.
   - `filas_completas`: cuántas filas acertaron las 4 columnas simultáneamente.
5. **Generar informe**:
   - `metricas_detalladas.csv`: métricas por archivo individual.
   - `resumen_por_estrategia.csv`: promedio de % por estrategia (pdfplumber vs easyocr).
   - `informe_comparativo.md`: resumen + detalle en Markdown.

### Métricas reportadas

| Métrica | Significado |
|---|---|
| `% item` | Tasa de acierto en nombre del analito |
| `% resultado` | Tasa de acierto en valor numérico |
| `% referencia` | Tasa de acierto en rango de referencia |
| `% flag` | Tasa de acierto en H/L |
| `% completo` | Fila entera correcta (4 columnas) |

## Resultados actuales

### pdfplumber (3 PDFs IDEXX)

- **100% en todas las columnas**: 24 filas × 4 columnas × 3 PDFs = 288 valores, todos correctos.
- Razón: El parser token-based es robusto para el formato predecible de IDEXX ProCyte Dx.

### EasyOCR (5 vouchers Vargas Vet)

| Columna | Pre-corrección (estimado) | Post-corrección |
|---|---|---|
| Item | ~20% | 40% |
| Resultado | ~4% | 20% |
| Referencia | ~23% | 34% |
| Flag | ~64% | 64% |
| Completo | ~3% | 12% |

- El corrector mejora significativamente Item (+20pp) y Referencia (+11pp).
- Flag ya estaba en 64% pre-corrección porque "H" y "L" son caracteres simples que EasyOCR acierta.
- Resultado sigue bajo (20%) porque el corrector no maneja errores numéricos ni caracteres faltantes.

## Pendiente / A mejorar

1. **Corrector de errores numéricos**: EasyOCR a menudo lee "9.42" como "9.42" (correcto) o "9.4?" (falta dígito). Se necesita un validador de formato numérico.
2. **Mejor detección de columnas**: Los gaps entre columnas varían por voucher; el header-detection falla si el encabezado no se lee bien.
3. **GPU**: `OCR_GPU = True` aceleraría EasyOCR si hay CUDA disponible.
4. **Más correcciones en el corrector**: A medida que se prueben más vouchers, se agregarán más correcciones.
5. **Análisis cualitativo en el informe**: Documentar los patrones de error típicos de EasyOCR.
